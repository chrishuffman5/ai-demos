01 · Concept

What is a hook?

A hook is a user-defined script, HTTP endpoint, or prompt that the CLI runs automatically at a specific point in its lifecycle — before a tool runs, when a session starts, when the agent tries to stop, and so on.

⚙ Outside the model

Hooks are plain code triggered by events, not text the model interprets. They run reliably regardless of how the agent was prompted.

⟳ Deterministic

Same event, same input, same outcome — every time. No "the model usually remembers to."

🔑 Your permissions

Hooks execute as you, with no extra sandbox. They can format, test, log, block, or inject context.

02 · Why they matter

Instruction → Enforcement

The essence of hooks is the shift from guiding the agent to guaranteeing behaviour. A rule in CLAUDE.md / AGENTS.md is a suggestion; a hook is a gate.

Custom instructions
# probabilistic
"Please don't run destructive
 commands like rm -rf."

Depends on the model choosing to comply.

PreToolUse hook
# deterministic
if cmd matches 'rm -rf':
    deny  # blocked, always

The command never executes — by construction.

03 · Structure

Anatomy of a hook

All three CLIs share the same three-level shape. You pick an event, narrow it with a matcher, and attach one or more handlers.

1
Event
The lifecycle moment that fires the hook — SessionStart, PreToolUse, Stop …
2
Matcher
Optional filter that narrows when it runs — e.g. tool name Bash, or regex Edit|Write.
3
Handler
What actually runs — a shell command, HTTP POST, MCP tool, or an LLM prompt that returns a decision.
Configuration is JSON (Claude Code, Copilot) or JSON / inline TOML (Codex). Multiple matching handlers for one event run in parallel.

05 · Handler types

Five ways a hook can run

Claude Code supports the full spectrum; Codex and Copilot are command-centric today.

command

Run a shell script. Input arrives on stdin as JSON; results return via exit code + stdout. All three CLIs.

http

POST the event JSON to a URL; the response body carries the decision. Claude Code (emerging in Copilot).

mcp_tool

Call a tool on a connected MCP server. Claude Code.

prompt

Send the event to a fast model for a single-turn yes/no judgment. Claude Code only.

agent

Spawn a subagent that can Read/Grep/Glob to verify a condition. Claude Code only (experimental).

⌁ Why it matters

Command = deterministic checks. Prompt / agent = semantic judgment. Choose by how much reasoning the gate needs.

06 · Shared vocabulary

The common event taxonomy

Despite different ecosystems, the three tools converge on a near-identical naming scheme — which makes hook knowledge portable.

SessionStart SessionEnd UserPromptSubmit PreToolUse PostToolUse Stop SubagentStop PreCompact Notification PermissionRequest errorOccurred
Naming varies slightly: Copilot uses camelCase (preToolUse, userPromptSubmitted); Claude Code & Codex use PascalCase. The control point is the same: PreToolUse is the universal blocking gate.
Tool 1 of 3

Claude Code

The most mature and extensive hook system — 20+ lifecycle events and all five handler types.

  • Config in layered JSON: ~/.claude/settings.json (user), .claude/settings.json (project), .local.json, plugins, and skill / agent frontmatter.
  • Matcher + if rule: filter by tool name, then by permission-rule syntax like Bash(git *).
  • Rich decision control: PreToolUse can allow / deny / ask / defer and even rewrite tool input before it runs.
  • Unique handlers: prompt & agent hooks delegate judgment to a model — "is this tool use appropriate?"

Tool 2 of 3

OpenAI Codex CLI

A Claude-style hooks engine, currently experimental and under active development.

  • Behind a feature flag: enable with [features] codex_hooks = true in config.toml.
  • Two config forms: hooks.json next to a config layer, or inline [[hooks.PreToolUse]] tables in TOML.
  • Locations: ~/.codex/ (user) and <repo>/.codex/ (project, must be trusted).
  • Command handlers only; hooks must be trusted to run; Windows support is temporarily disabled.
Separately, Codex has a notify setting — a command that receives a JSON payload for desktop notifications, independent of the hooks engine.
Codex · events

Supported events

The documented, working set today — turn-scoped and session-scoped:

SessionStart

Matcher = startup | resume. Inject developer context via stdout or additionalContext.

UserPromptSubmit

Add context or block the prompt. Matcher not used.

PreToolUse ⚑

Can deny a command. Currently intercepts Bash only — a guardrail, not a complete boundary.

PostToolUse

Bash results only. block replaces the tool result with feedback and continues.

Stop

decision:"block" doesn't reject — it spins up a continuation prompt to keep working.

SubagentStart

Fires at subagent-start scope. (Advanced config also references PermissionRequest & Pre/PostCompact.)

Tool 3 of 3

GitHub Copilot CLI

A repository-native hook system shared between Copilot CLI and the cloud Coding Agent — the repo is the policy boundary.

  • Config in .github/hooks/*.json with "version": 1. CLI loads from your working directory; the cloud agent reads from the default branch.
  • Command-only (type: "command"), committed to git and reviewed like any other code.
  • Cross-platform by design: each handler carries a bash key and a powershell key (pwsh 7+ on Windows).
  • Sequential execution; the first non-zero exit halts the chain.
Copilot CLI · events

Supported events

Six documented events. Only preToolUse can block — every other event is observational.

sessionStart

New or resumed session. Init env, validate state, set up resources. Observational.

sessionEnd

Session completes / terminates. Cleanup, archive logs, notify. Observational.

userPromptSubmitted

User submits a prompt. Audit & usage analysis. Cannot block.

preToolUse ⚑

The control hook. Returns permissionDecision: allow / deny / ask.

postToolUse

After a tool returns. Log results, scan outputs. Observational.

errorOccurred

Fires on error. Forensics & alerting. Observational.

A broader 8-event set (adding Stop, SubagentStart/Stop, PreCompact) is appearing in the VS Code preview; the six above are the GA Copilot CLI surface.

08 · Communication

How a hook talks back

Two channels: simple exit codes, or structured JSON on stdout for fine-grained control.

Exit codes
0  → success; stdout may add context
2  → blocking error; stderr → model
*  → non-blocking error, continue

Note: only exit 2 blocks — exit 1 is treated as a non-blocking warning.

JSON decision (PreToolUse)
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason":
        "Destructive command blocked"
  }
}

Decisions: allow · deny · ask · defer (+ rewrite input on Claude Code).

10 · Example B

PreToolUse in practice

The universal control gate — inspect the pending tool call and deny it before anything runs.

Block rm -rf / DROP TABLE Protect secrets & .env files Require review for prod ops Prompt-judge risky calls (Claude)
Copilot CLI — deny destructive Bash (preToolUse script)
#!/bin/bash
INPUT=$(cat); CMD=$(echo "$INPUT" | jq -r '.toolArgs.command')
if echo "$CMD" | grep -qE 'rm\s+-rf'; then
  echo '{"permissionDecision":"deny","permissionDecisionReason":"rm -rf blocked"}'
else echo '{"permissionDecision":"allow"}'; fi
Claude Code — semantic gate via prompt hook
{ "type": "prompt", "prompt": "Evaluate if this tool use is safe: $TOOL_INPUT", "timeout": 30 }
11 · Example C

Stop in practice

Fires when the agent thinks it's finished. Block it to enforce a quality gate, or send it back to keep working.

Gate completion until tests pass Force another pass on failures Notify when truly done
Claude Code — don't stop while tests are red (exit-code form)
#!/bin/bash  # Stop hook
if ! npm test --silent; then
  echo "Tests failing — keep going." >&2
  exit 2   # blocks the stop; conversation continues
fi
Codex — Stop "block" creates a continuation prompt (JSON on stdout)
{ "decision": "block", "reason": "Run one more pass over the failing tests." }
In Codex, a blocked Stop turns the reason into a fresh user prompt; in Claude Code, exit 2 simply continues the turn.
12 · Operate safely

Security & best practices

  • No sandbox. Hooks run with your full user permissions. Treat them like production code and review every line.
  • Use exit 2 to enforce. A policy hook that exits 1 fails open — the action proceeds anyway.
  • Keep them small & fast. They run on every matching event; a slow hook stalls the whole session.
  • Commit team guardrails to project config (Claude) or the repo's default branch (Copilot) so everyone shares the same gates.
  • Mind the trust model. Codex requires hooks to be trusted; project-local hooks load only for trusted layers.
  • Validate & quote input. Parse the stdin JSON with jq; never interpolate raw tool args into a shell.
Golden rule: write what the agent may do as code. A hook that returns deny is certainty; a sentence in an instructions file is a hope.
Wrap-up

Takeaways

Claude Code

Richest system: 20+ events, 5 handler types, prompt/agent judgment, allow/deny/ask/defer + input rewrite. The reference implementation.

Codex CLI

Experimental & config-native (TOML/JSON). Command hooks, Bash-focused PreToolUse, Stop-as-continuation. Trust-gated, no Windows yet.

Copilot CLI

Repo-native & team-shared via git. 6 events, command-only, cross-platform bash+pwsh. Only preToolUse blocks.

  • Same mental model everywhere: event → matcher → handler, talking back via exit codes & JSON.
  • PreToolUse is the universal control point; SessionStart and Stop are the universal context + completion points.
  • Hooks turn intent into enforcement — the deterministic shell around a probabilistic agent.
