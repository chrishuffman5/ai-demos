One standard for customizing GitHub Copilot — and every other agent CLI we run. Stop scattering rules across vendor-specific files.

Conventions, always on Capabilities, on demand
The problem

Customization sprawl & vendor lock-in

Copilot alone gives us four overlapping ways to inject rules:

  • .github/copilot-instructions.md — repo-wide, always on
  • .github/instructions/*.instructions.md — path-scoped
  • .github/prompts/*.prompt.md — reusable slash commands
  • .github/agents/*.agent.md — personas & handoffs

Two pains: they drift repo-to-repo, and most are Copilot/VS Code-only. We also run claude, codex, and copilot CLI — we need a layer all of them read.

The standard

Two layers. That's it.

Layer 1 · Always on

AGENTS.md

How we build here. Conventions, stack versions, build/test commands, guardrails — loaded on every request.

"Use pwsh 7. Python ≥ 3.13. Never commit secrets."
Layer 2 · On demand

Skills

How to do task X. Packaged, repeatable capability with bundled scripts & references — loaded only when relevant.

"Run a Splunk→LogParq migration the way we do it."

Conventions are always on. Capabilities load when needed.

Layer 1

AGENTS.md — the README for agents

  • Replaces .github/copilot-instructions.md — same job, broader reach.
  • Read by Copilot coding agent, CLI, and VS Code agent mode — plus Claude, Codex & others via the shared standard.
  • Nested files supported: the nearest AGENTS.md in the tree wins → clean per-package scoping in monorepos.
Migrate, don't duplicate. If both AGENTS.md and copilot-instructions.md exist at the root, the CLI loads both — they stack, they don't override. Move the content into AGENTS.md and delete the old file.
Layer 1 · example

A minimal AGENTS.md

# AGENTS.md — repo root

## Stack
- PowerShell 7+ (pwsh). Python ≥ 3.13. .NET 8 / C# 12. Node ≥ 24.
- Databases: Postgres ≥ 16, SQL Server 2022, DuckDB ≥ 1.5.

## Conventions
- No aliases in committed PowerShell. Approved verbs only.
- Async all the way in C#; never .Result / .Wait().
- All docs & PRDs in Markdown. Never generate Office files.

## Build & validate
- Lint:  pwsh ./build/lint.ps1
- Test:  pwsh ./build/test.ps1
- Run these before proposing changes. Trust this file; only
  explore if it's incomplete or wrong.

Keep it tight — conventions, versions, commands, guardrails. Not a wiki.

Layer 2

Skills — packaged, portable capability

  • A folder with a SKILL.md (frontmatter: name + description) plus optional scripts/, references/, assets/.
  • In Copilot: .github/skills/<name>/ (repo) or ~/.copilot/skills/ (personal).
  • Progressive disclosure: Copilot reads name + description first, loads the full instructions only when relevant, and pulls bundled resources on demand.
  • Portable: the open Agent Skills standard — the same skill runs in VS Code, Copilot CLI, the coding agent, and Claude / Codex.
Layer 2 · anatomy

What a Skill looks like

.github/skills/logparq-migration/
├── SKILL.md          # required: frontmatter + instructions
├── references/
│   └── schema-map.md   # loaded on demand
└── scripts/
    └── export.ps1      # bundled, executable

# SKILL.md frontmatter
---
name: logparq-migration
description: Migrate Splunk indexes to the DuckDB/Parquet
  LogParq tiered layout. Use when asked to move logging
  off Splunk or build a Parquet log pipeline.
---

The description is the trigger — write it as "what + when to use."

The mapping

Copilot todayMove to
copilot-instructions.md → AGENTS.mdAlways-on conventions. Direct replacement.
*.prompt.md → SkillPure upgrade — adds scripts, references, portability.
*.instructions.md (applyTo) → Skill — or keepKeep when a rule must deterministically fire on a file glob.
*.agent.md → subagent (separate topic)Persona/handoffs — not a Skill, not in scope here.
Determinism matters

When to keep applyTo instructions

Skills trigger on a semantic match — the model decides the description fits. applyTo triggers on a file glob — it fires every time, guaranteed. If a rule must apply whenever a path is touched, keep it as an instructions file.

--- .github/instructions/terraform.instructions.md ---
applyTo: "**/*.tf,**/*.tfvars"
description: Terraform standards
---
- Pin provider versions. No hardcoded secrets.
- Every resource carries the standard tag block.
- Run terraform fmt + validate before changes.

Keep globs narrow and non-overlapping — Copilot resolves conflicts non-deterministically. Anything that applies everywhere belongs in AGENTS.md, not a ** glob.

Where does this rule go?

Applies to every request, in every repo?
AGENTS.md
Repeatable multi-step task, needs scripts, or must work cross-tool?
Skill
Must deterministically fire on a file pattern?
.instructions.md
One-off chat task you might rerun occasionally?
prompt — or just ask
Why this wins

One source of truth, zero lock-in

  • AGENTS.md + Skills are the only layer read across claude, codex, and copilot.
  • Skills follow an open standard — write once, run on every compatible agent.
  • No re-authoring rules per vendor. No drift between what each tool "knows."
  • New hire opens any repo → the agent already knows how we build.

The pitch isn't "Skills are more powerful." It's "instructions and prompt files lock us into one vendor's CLI."

Do this now

Rollout checklist

  1. Add AGENTS.md at each repo root. Move copilot-instructions.md content in, then delete the old file.
  2. Convert *.prompt.md → Skills under .github/skills/<name>/. Write descriptions as "what + when."
  3. Keep only glob-deterministic rules as *.instructions.md. Fold the rest into AGENTS.md.
  4. Don't over-skill: one-line conventions live in AGENTS.md, not a micro-skill.
  5. Confirm your IDE surface reads AGENTS.md before deleting (VS Code agent mode, CLI, coding agent ✓).
